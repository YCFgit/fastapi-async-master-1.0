## 1. 系统概述

AsyncTaskFlow 是一个**通用 API 任务网关**。它的核心理念是：

> 你只需要注册一个 HTTP API 的调用方式（任务类型），系统就能异步地、可靠地帮你调用这个 API，处理重试、限流、熔断等复杂逻辑。

### 1.1 核心架构

```
用户提交任务 → Redis 队列 → Celery Worker 消费 → 调用外部 API → 返回结果
```

系统由 6 个 Docker 容器组成：

| 容器 | 作用 | 端口 |
|------|------|------|
| `redis` | 数据存储 + 消息队列 | 6379 |
| `api` | FastAPI REST 接口 | 8000 |
| `frontend` | React 前端界面 | 3000 |
| `worker` × 3 | Celery 任务执行器 | - |
| `scheduler` | Celery Beat 调度器 | - |

### 1.2 任务生命周期

一个任务从提交到完成会经历以下状态：

```
PENDING（等待）→ ACTIVE（执行中）→ COMPLETED（完成）
                    ↓ 失败
                FAILED（失败）→ SCHEDULED（等待重试）→ PENDING
                    ↓ 重试耗尽
                DLQ（死信队列）
```

**举例说明：** 假设你注册了一个"文本翻译"的 API。提交一个翻译任务后：
1. 任务状态变为 `PENDING`，进入主队列等待
2. Worker 消费任务，调用翻译 API，状态变为 `ACTIVE`
3. API 返回翻译结果，状态变为 `COMPLETED`
4. 如果 API 返回 500 错误，状态变为 `FAILED`，然后进入 `SCHEDULED` 等待重试
5. 重试 3 次仍然失败，状态变为 `DLQ`（死信），需要人工介入

## 2. 快速开始

### 2.1 安装与启动

```bash
# 1. 进入项目目录
cd fastapi-async-master-1.0

# 2. 复制环境变量配置文件
cp .env.example .env

# 3. 编辑 .env 文件（可选，使用默认值即可启动）
vim .env

# 4. 使用 Docker Compose 启动所有服务
docker compose up -d --build

# 5. 查看启动状态
docker compose ps
```

### 2.2 访问系统

启动成功后，你可以访问以下地址：

| 地址 | 说明 |
|------|------|
| `http://localhost:3000` | **前端界面**（主要操作入口） |
| `http://localhost:8000/docs` | **API 文档**（Swagger UI，自动生成） |
| `http://localhost:8000/health` | **健康检查**（查看系统状态） |

### 2.3 验证系统正常运行

打开浏览器访问 `http://localhost:3000`，你应该看到：
- 顶部状态栏显示 **SYSTEM ONLINE**（绿色圆点闪烁）
- 仪表盘页面展示队列统计和任务流程图
- Worker 状态面板显示 3 个活跃的工作节点

## 3. 前端界面详解

### 3.1 语言切换

系统支持**中英文切换**。点击右上角的 🌐 Globe 图标按钮即可切换语言。语言偏好会自动保存到浏览器本地存储。

- 英文模式下按钮显示 **中文**
- 中文模式下按钮显示 **EN**

### 3.2 导航栏

顶部导航栏包含 4 个页面：

| 图标 | 页面 | 说明 |
|------|------|------|
| ⚡ | **仪表盘** | 系统总览，快速提交任务 |
| ⚙️ | **任务类型** | 注册和管理 API 任务类型 |
| 🕐 | **历史记录** | 查看所有任务的执行记录 |
| 🗑️ | **清理管理** | 数据清理工具（即将推出） |

### 3.3 仪表盘页面

仪表盘是系统的"驾驶舱"，分为 4 个区域：

#### 3.3.1 快速提交任务

页面顶部的提交表单用于快速提交一个任务：

**操作步骤：**
1. **选择任务类型**（下拉框）：选择你之前注册的任务类型，例如 `translate`
2. **输入内容**（文本框）：输入要处理的内容，例如 `Hello, world!`
3. **输入参数**（可选，JSON 格式）：附加参数，例如 `{"target_lang": "zh"}`
4. **点击 Execute 按钮**：提交任务

**示例：** 翻译任务
```
类型: translate
内容: Hello, world!
参数: {"target_lang": "zh"}
```

提交成功后，页面右上角会弹出提示，显示新任务的 ID。

#### 3.3.2 队列统计

显示 4 个队列的深度（当前积压任务数）：

| 队列 | 图标 | 说明 |
|------|------|------|
| **PRIMARY** | 📥 | 主队列，等待执行的任务 |
| **RETRY** | 🔄 | 重试队列，执行失败等待重试的任务 |
| **SCHEDULED** | ⏰ | 调度队列，正在等待重试延迟的任务 |
| **DLQ** | 💀 | 死信队列，重试耗尽的失败任务 |

下方还有一个**状态分布**条形图，直观展示各状态任务的占比。

#### 3.3.3 任务流程图

一个可视化流程图，展示任务从提交到完成的完整流转路径：

```
[任务提交] → [主队列] → [执行中] → [已完成]
                           ↓
                        [失败] → [死信队列]
                           ↓
                        [调度队列] → [重试队列] → [执行中]
```

**交互说明：**
- 每个节点显示对应的任务数量
- 点击有数量的节点，会跳转到历史记录页面并自动筛选
- 可以拖拽和缩放流程图

#### 3.3.4 Worker 状态

显示所有 Celery Worker 节点的实时状态：

- **节点名称**：Worker 的 hostname-PID
- **状态标签**：HEALTHY（健康）/ STALE（过期）/ NO_HEARTBEAT（无心跳）
- **熔断器状态**：CLOSED（正常）/ HALF-OPEN（半开）/ OPEN（断开）
- **成功/失败计数**：该节点处理任务的统计数据
- **最后活跃时间**：最后一次心跳的时间

## 4. 任务类型管理

任务类型是 AsyncTaskFlow 的核心概念。它定义了"如何调用一个外部 API"。

### 4.1 创建任务类型

**操作步骤：**
1. 点击左侧导航栏的 **任务类型**
2. 点击右上角的 **注册新类型** 按钮
3. 在弹出的对话框中填写配置
4. 点击 **创建任务类型** 保存

#### 4.1.1 基本信息

| 字段 | 必填 | 说明 | 示例 |
|------|------|------|------|
| `type_id` | 是 | 唯一标识符，只能用字母、数字、下划线 | `translate` |
| `name` | 是 | 人类可读的名称 | `文本翻译` |
| `description` | 否 | 描述信息 | `调用翻译API翻译文本` |

#### 4.1.2 API 配置

| 字段 | 必填 | 说明 | 示例 |
|------|------|------|------|
| `api_base_url` | 是 | API 的基础 URL | `https://api.mymemory.translated.net` |
| `api_endpoint` | 是 | API 的端点路径 | `/get` |
| `http_method` | 否 | HTTP 方法，默认 POST | `GET` |

#### 4.1.3 认证配置

系统支持 4 种认证方式：

**方式一：无认证（NONE）**
- 不需要额外配置
- 适用于公开 API

**方式二：Bearer Token**
- 填写 Bearer Token 字段
- 系统会自动添加请求头 `Authorization: Bearer <token>`
- 示例：`sk-abc123xyz`
- 支持环境变量：`${OPENAI_API_KEY}`

**方式三：API Key**
- 填写 Header Name 和 Header Value
- 示例：Header Name = `X-API-Key`，Header Value = `my-secret-key`
- 支持环境变量：`${MY_API_KEY}`

**方式四：Basic Auth**
- 填写 Username 和 Password
- 系统会自动添加请求头 `Authorization: Basic <base64>`

#### 4.1.4 模板与响应

**请求模板（Jinja2）**

使用 Jinja2 模板语法定义请求体。可用变量：

| 变量 | 说明 | 示例用法 |
|------|------|---------|
| `{{content}}` | 任务提交时的内容 | `{"text": "{{content}}"}` |
| `{{params.xxx}}` | 任务提交时的附加参数 | `{{params.target_lang}}` |
| `{{task_id}}` | 任务的唯一 ID | `{"request_id": "{{task_id}}"}` |

**示例：POST 请求模板**
```json
{
  "q": "{{content}}",
  "langpair": "en|{{params.target_lang | default('zh')}}"
}
```

**示例：GET 请求（无模板）**
- 如果使用 GET 方法，可以不填写请求模板
- 参数会自动拼接到 URL 上

**响应提取（JSONPath）**

使用 JSONPath 语法从 API 响应中提取结果：

| API 响应 | JSONPath 表达式 | 提取结果 |
|----------|----------------|---------|
| `{"data": {"result": "你好"}}` | `$.data.result` | `你好` |
| `{"choices": [{"text": "回答"}]}` | `$.choices[0].text` | `回答` |
| `{"translation": "翻译结果"}` | `$.translation` | `翻译结果` |

#### 4.1.5 重试与限流

| 字段 | 说明 | 默认值 | 示例 |
|------|------|--------|------|
| `max_retries` | 最大重试次数 | 3 | `5` |
| `rate_limit_requests` | 限流：时间窗内最大请求数 | 不限 | `10` |
| `rate_limit_interval` | 限流：时间窗（秒） | 不限 | `60` |

**限流示例：** 设置 `rate_limit_requests=10`，`rate_limit_interval=60` 表示每 60 秒最多处理 10 个该类型的任务。

### 4.2 完整示例：注册 OpenAI 翻译任务

**步骤 1：填写基本信息**
```
type_id:    openai_translate
name:       OpenAI 文本翻译
description: 使用 GPT 模型翻译文本
```

**步骤 2：填写 API 配置**
```
api_base_url:  https://api.openai.com
api_endpoint:  /v1/chat/completions
http_method:   POST
```

**步骤 3：配置认证**
```
认证类型: Bearer Token
Token:    ${OPENAI_API_KEY}
```

**步骤 4：填写请求模板**
```json
{
  "model": "gpt-4o-mini",
  "messages": [
    {"role": "system", "content": "You are a translator. Translate the user's text to {{params.target_lang | default('Chinese')}}. Only output the translation."},
    {"role": "user", "content": "{{content}}"}
  ]
}
```

**步骤 5：填写响应提取**
```
response_jsonpath: $.choices[0].message.content
```

**步骤 6：配置重试**
```
max_retries: 3
```

**步骤 7：点击"创建任务类型"保存**

### 4.3 测试任务类型

创建完成后，可以点击列表中的 **测试** 按钮验证配置是否正确。

测试会发送一个模拟请求到配置的 API，返回：
- **状态**：成功 / 失败
- **HTTP 状态码**：如 200, 401, 500
- **响应时间**：毫秒
- **请求内容**：实际发送的请求体
- **提取结果**：从响应中提取的结果
- **响应体**：完整的 API 响应
- **错误信息**：如果失败，显示错误原因

**常见测试失败原因：**
- `401 Unauthorized`：认证信息错误，检查 Token 或 API Key
- `404 Not Found`：API 端点路径错误
- `Connection refused`：API 地址不可达
- `JSONPath extraction failed`：响应提取表达式错误

### 4.4 编辑与删除任务类型

- **编辑**：点击列表中的 **编辑** 按钮，修改配置后保存
- **删除**：点击列表中的 🗑️ 按钮，确认后删除（不可恢复）
- **停用/启用**：点击 **停用** 或 **启用** 按钮，暂停或恢复任务类型的使用

停用后：
- 已提交的该类型任务会继续执行
- 无法提交新的该类型任务

### 4.5 通过 API 注册任务类型

也可以直接调用 API 注册：

```bash
curl -X POST http://localhost:8000/api/v1/task-types \
  -H "Content-Type: application/json" \
  -d '{
    "type_id": "translate",
    "name": "Text Translation",
    "api_base_url": "https://api.mymemory.translated.net",
    "api_endpoint": "/get",
    "http_method": "GET",
    "request_template": "",
    "response_jsonpath": "$.responseData.translatedText",
    "max_retries": 3,
    "enabled": true
  }'
```

## 5. 提交任务

### 5.1 通过前端提交

在仪表盘页面的顶部表单：

1. 从下拉框选择任务类型
2. 在内容框输入要处理的内容
3. （可选）在参数框输入 JSON 格式的附加参数
4. 点击 **Execute** 按钮

**示例：提交翻译任务**
```
类型: translate
内容: The quick brown fox jumps over the lazy dog.
参数: {"target_lang": "zh"}
```

提交成功后会显示：`Task submitted: <task_id>`

### 5.2 通过 API 提交

```bash
# 基本提交
curl -X POST http://localhost:8000/api/v1/tasks/submit \
  -H "Content-Type: application/json" \
  -d '{
    "task_type": "translate",
    "content": "Hello, world!"
  }'

# 带参数和优先级
curl -X POST http://localhost:8000/api/v1/tasks/submit \
  -H "Content-Type: application/json" \
  -d '{
    "task_type": "translate",
    "content": "Hello, world!",
    "params": {"target_lang": "zh"},
    "priority": 5
  }'

# 带回调 URL
curl -X POST http://localhost:8000/api/v1/tasks/submit \
  -H "Content-Type: application/json" \
  -d '{
    "task_type": "translate",
    "content": "Hello, world!",
    "callback_url": "https://myapp.com/webhook/task-done"
  }'
```

### 5.3 提交参数说明

| 参数 | 必填 | 类型 | 说明 | 示例 |
|------|------|------|------|------|
| `task_type` | 是 | string | 任务类型的 type_id | `"translate"` |
| `content` | 是 | string | 要处理的内容 | `"Hello"` |
| `params` | 否 | object | 附加参数，在模板中用 `{{params.xxx}}` 引用 | `{"lang": "zh"}` |
| `priority` | 否 | int(0-10) | 优先级，数字越大越优先，默认 0 | `5` |
| `callback_url` | 否 | string | 任务完成后的回调 URL | `"https://..."` |

### 5.4 优先级说明

- 默认优先级为 0（普通）
- 优先级范围 0-10，数字越大越优先
- 高优先级任务会优先被 Worker 消费
- 使用 Redis Sorted Set 实现，分数为优先级的负值

**示例：** 同时提交两个任务
```
任务A: priority=0, content="普通任务"
任务B: priority=10, content="紧急任务"
```
任务 B 会优先被处理。

## 6. 任务历史管理

### 6.1 查看任务列表

点击导航栏的 **历史记录** 进入任务历史页面。

**功能说明：**
- **搜索框**：输入任务 ID 进行模糊搜索
- **状态筛选**：下拉框选择状态（PENDING / ACTIVE / COMPLETED / FAILED / SCHEDULED / DLQ）
- **类型筛选**：下拉框选择任务类型
- **分页**：底部的上一页/下一页按钮

#### 表格字段说明

| 列名 | 说明 |
|------|------|
| **TASK_ID** | 任务的唯一标识（UUID） |
| **STATUS** | 当前状态，用不同颜色标识 |
| **TYPE** | 任务类型的 type_id |
| **CREATED** | 创建时间 |
| **ACTIONS** | 操作按钮 |

#### 状态颜色对照

| 状态 | 颜色 | 含义 |
|------|------|------|
| PENDING | 灰色 | 等待执行 |
| ACTIVE | 绿色 | 正在执行 |
| COMPLETED | 蓝色 | 已完成 |
| FAILED | 橙色 | 执行失败（等待重试） |
| SCHEDULED | 紫色 | 已调度（等待重试延迟） |
| DLQ | 红色 | 死信（重试耗尽） |

### 6.2 查看任务详情

点击列表中的 **查看** 按钮，展开任务的详细信息：

#### 6.2.1 任务信息

- **类型**：任务类型的 type_id
- **内容**：提交时的内容
- **有结果**：是否已产生结果
- **结果**：任务执行的结果数据

#### 6.2.2 时间信息

- **创建**：任务创建时间
- **更新**：最后更新时间
- **完成**：任务完成时间（如果已完成）
- **耗时**：从创建到完成的耗时

#### 6.2.3 重试信息

- **重试**：当前重试次数 / 最大重试次数
- **重试时间**：下次重试的计划时间（如果在调度队列中）

#### 6.2.4 参数

显示提交时的附加参数（JSON 格式）。

#### 6.2.5 状态变更历史

按时间顺序显示任务经历的每一次状态变更。

#### 6.2.6 错误历史

如果任务执行失败，显示所有错误记录，包括：
- 错误发生的时间
- 错误信息（如 HTTP 状态码、网络超时等）

#### 6.2.7 任务结果

如果任务执行成功，显示从外部 API 提取的结果数据。

### 6.3 删除任务

1. 点击列表中的 **删除** 按钮
2. 在弹出的确认对话框中确认
3. 任务将从所有 Redis 队列、状态和统计中永久移除

**注意：** 删除操作不可撤销。

## 7. 队列与状态监控

### 7.1 队列深度

仪表盘上的 4 个队列卡片实时显示：

| 队列 | 正常值 | 警告信号 | 说明 |
|------|--------|---------|------|
| PRIMARY | 0-100 | > 500 | 主队列积压严重，可能需要增加 Worker |
| RETRY | 0-50 | > 200 | 重试任务过多，外部 API 可能有问题 |
| SCHEDULED | 0-50 | > 200 | 等待重试的任务过多 |
| DLQ | 0 | > 0 | 有任务失败，需要关注 |

### 7.2 状态分布

条形图展示各状态任务的占比，正常情况下：
- 大部分应该是 COMPLETED（蓝色）
- 少量 PENDING（灰色）
- 极少 FAILED / DLQ

### 7.3 SSE 实时推送

仪表盘通过 SSE（Server-Sent Events）实时接收状态更新。如果连接断开：
- 页面顶部会显示 **CONNECTION_LOST** 警告
- 系统会自动尝试重连（每 5 秒）
- 重连成功后自动恢复实时更新

## 8. Worker 节点监控

### 8.1 Worker 状态面板

仪表盘底部的 Worker 状态面板显示：

**总体统计：**
- **TOTAL_WORKERS**：Worker 总数
- **HEALTHY**：健康节点数
- **STALE**：过期节点数
- **整体状态**：HEALTHY / UNHEALTHY

**每个节点的详情：**

| 字段 | 说明 |
|------|------|
| 节点名称 | hostname-PID 格式 |
| 状态 | HEALTHY / STALE / NO_HEARTBEAT |
| 熔断器 | CLOSED（正常）/ HALF-OPEN（半开）/ OPEN（断开） |
| 成功数 | 处理成功的任务数 |
| 失败数 | 处理失败的任务数 |
| 最后活跃 | 最后一次心跳时间 |

### 8.2 熔断器说明

熔断器是一种保护机制，防止持续调用已经出问题的外部 API：

| 状态 | 说明 | 行为 |
|------|------|------|
| **CLOSED** | 正常状态 | 所有任务正常执行 |
| **OPEN** | 熔断状态 | 暂停执行该类型的任务，避免无意义的重试 |
| **HALF-OPEN** | 半开状态 | 尝试执行少量任务，如果成功则恢复 CLOSED |

**触发条件：** 连续失败次数达到 `circuit_breaker_fail_max`（默认 10 次）

**自动恢复：** 等待 `circuit_breaker_reset_timeout`（默认 120 秒）后进入 HALF-OPEN

### 8.3 手动操作

通过 API 可以手动控制熔断器：

```bash
# 重置熔断器（恢复正常）
curl -X POST "http://localhost:8000/api/v1/workers/reset-circuit-breaker?type_id=translate"

# 手动打开熔断器（暂停该类型任务）
curl -X POST "http://localhost:8000/api/v1/workers/open-circuit-breaker?type_id=translate"
```

## 11. 常见问题与排查

### 11.1 任务一直停留在 PENDING 状态

**可能原因：**
1. Worker 没有启动或已崩溃
2. Consumer 进程没有运行
3. Redis 连接异常

**排查步骤：**
```bash
# 1. 检查 Worker 容器是否运行
docker compose ps worker

# 2. 查看 Worker 日志
docker compose logs worker --tail=50

# 3. 检查 Redis 连接
curl http://localhost:8000/api/v1/redis/health

# 4. 查看健康检查
curl http://localhost:8000/health
```

### 11.2 任务一直失败进入 DLQ

**可能原因：**
1. 外部 API 地址错误
2. 认证信息过期或错误
3. API 返回非预期格式
4. 网络不通

**排查步骤：**
```bash
# 1. 查看任务详情中的错误信息
curl http://localhost:8000/api/v1/tasks/{task_id}

# 2. 测试任务类型配置
curl -X POST http://localhost:8000/api/v1/task-types/{type_id}/test

# 3. 检查外部 API 是否可达
curl -v https://your-api.example.com/health
```

### 11.3 Worker 状态显示 STALE

**说明：** Worker 心跳超过 60 秒没有更新。

**可能原因：**
1. Worker 进程卡死（如外部 API 响应极慢）
2. Worker 内存超限被 OOM Kill
3. 网络分区

**解决方法：**
```bash
# 重启 Worker
docker compose restart worker
```

### 11.4 熔断器打开（OPEN）

**说明：** 该类型的任务连续失败 10 次，系统自动暂停执行。

**解决方法：**
```bash
# 1. 先排查外部 API 问题
# 2. 修复后，手动重置熔断器
curl -X POST "http://localhost:8000/api/v1/workers/reset-circuit-breaker?type_id=your_type_id"
```

### 11.5 前端显示 CONNECTION_LOST

**说明：** SSE 实时连接断开。

**可能原因：**
1. API 容器崩溃
2. Nginx 代理配置问题
3. 浏览器网络中断

**排查步骤：**
```bash
# 1. 检查 API 容器
docker compose ps api

# 2. 查看 API 日志
docker compose logs api --tail=50

# 3. 手动测试 SSE
curl -N http://localhost:8000/api/v1/queues/status/stream
```

### 11.6 如何重置所有数据

```bash
# 使用内置的 reset 工具
docker compose --profile tools run reset

# 或者手动清空 Redis
docker compose exec redis redis-cli FLUSHDB
```

### 11.7 如何查看详细日志

```bash
# 查看所有服务日志
docker compose logs -f

# 只看 API 日志
docker compose logs -f api

# 只看 Worker 日志（3 个副本）
docker compose logs -f worker

# 只看最近 100 行
docker compose logs --tail=100 worker
```

### 11.8 如何扩展 Worker 数量

编辑 `.env` 文件：
```bash
WORKER_REPLICAS=5
```

然后重启：
```bash
docker compose up -d --build worker
```

## 附录：术语表

| 术语 | 说明 |
|------|------|
| **Task Type** | 任务类型，定义了如何调用一个外部 API |
| **Task** | 任务，一次具体的 API 调用请求 |
| **Consumer** | 消费者，从 Redis 队列中取出任务分发给 Worker |
| **Worker** | 工作节点，Celery Worker，执行具体的 HTTP 调用 |
| **Scheduler** | 调度器，Celery Beat，负责将到期的重试任务移回主队列 |
| **Primary Queue** | 主队列，新任务和重试到期任务的入口 |
| **Retry Queue** | 重试队列，执行失败等待重试的任务 |
| **Scheduled Queue** | 调度队列，正在等待重试延迟的任务 |
| **DLQ** | Dead Letter Queue（死信队列），重试耗尽的失败任务 |
| **Circuit Breaker** | 熔断器，防止持续调用已出问题的外部 API |
| **Rate Limiter** | 限流器，控制对特定 API 的调用频率 |
| **JSONPath** | JSON 路径表达式，用于从 API 响应中提取数据 |
| **Jinja2** | Python 模板引擎，用于生成请求体 |
| **SSE** | Server-Sent Events，服务端推送实时更新 |
| **SSRF** | Server-Side Request Forgery，服务端请求伪造防护 |

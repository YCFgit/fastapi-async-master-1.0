# AsyncTaskFlow v1.0 — 项目综合介绍文档

> **通用 API 任务网关** — 将任意 HTTP API 调用转化为可管理、可观测、可重试的异步任务

---

## 一、项目概述

### 1.1 什么是 AsyncTaskFlow？

AsyncTaskFlow 是一个**通用 API 任务网关**（Generic API Task Gateway），用于异步、可靠地调用外部 HTTP API。用户只需注册一个 API 的调用方式（任务类型），系统便能自动处理任务调度、重试、限流、熔断等复杂逻辑。

**一句话概括：** 你告诉它"怎么调 API"，它帮你"可靠地、异步地、可观察地"调 API。

### 1.2 核心理念

将任意 HTTP API 调用转化为**可管理、可观测、可重试**的异步任务。

- **可管理** — 任务类型注册、优先级调度、限流控制、熔断保护
- **可观测** — 实时状态推送（SSE）、任务状态历史、错误记录、队列监控
- **可重试** — 自动重试机制、延迟重试、死信队列（DLQ）、手动重试

### 1.3 适用场景

| 场景 | 说明 |
|------|------|
| AI 模型调用 | 翻译、文本生成、图像处理等 AI API 调用 |
| 第三方 API 集成 | 统一管理多个外部 API 的调用逻辑 |
| 批量数据处理 | 大量数据需要异步处理的场景 |
| 微服务通信 | 服务间的异步消息传递 |
| 定时任务 | 需要延迟执行或定时重试的任务 |

### 1.4 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| **后端框架** | FastAPI (Python) | 高性能异步 Web 框架 |
| **任务队列** | Celery | 分布式任务队列 |
| **消息代理** | Redis 7 | 数据存储 + 消息队列 + 缓存 |
| **前端框架** | React 19 + TypeScript | 现代化 UI 框架 |
| **构建工具** | Vite | 快速前端构建 |
| **样式框架** | Tailwind CSS | 原子化 CSS 框架 |
| **流程图** | ReactFlow | 交互式流程图可视化 |
| **部署方案** | Docker Compose | 容器化一键部署 |

```
                    │   API Server    │

                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
     ┌────────▼───────┐  ┌──▼──┐  ┌───────▼───────┐
     │   Redis 7      │  │     │  │  Celery Beat   │
     │ 数据存储+队列   │  │     │  │   (调度器)     │
```

### 2.2 容器组成

| 容器 | 作用 | 端口 | 镜像 |
|------|------|------|------|
| `redis` | 数据存储 + 消息队列（RDB + AOF 双重持久化） | 6379 | redis:7-alpine |
| `api` | FastAPI REST 接口 | 8000 | 自构建 |
| `frontend` | React 前端界面（Nginx 静态服务） | 3000 | 自构建 |
| `worker` × 3 | Celery 任务执行器 | — | 自构建 |
| `scheduler` | Celery Beat 调度器 | — | 自构建 |
| `reset` | 数据重置工具（按需激活） | — | 自构建 |

### 2.3 数据流

```
用户提交任务
    │
    ▼
API Server 写入 Redis Sorted Set（优先级队列）
    │
    ▼
Consumer 从队列拉取任务
    │
    ▼
Celery Worker 执行任务
    │
    ├─→ 调用外部 HTTP API
    │
    ├─→ 提取响应结果（JSONPath）
    │
    ├─→ 写回 Redis（状态 + 结果）
    │
    └─→ 触发回调通知（Webhook）
```

### 2.4 任务生命周期

```
                         ┌──────────────────────────────────────────┐
                         │                                          │
  提交任务 ──────▶ PENDING ──────▶ ACTIVE ──────▶ COMPLETED        │
                         │                  │                       │
                         │                  │ (临时错误: 429/5xx)   │
                         │                  ▼                       │
                         │              SCHEDULED ──(延迟到期)──▶ PENDING
                         │                  │                       │
                         │                  │ (永久错误: 4xx)       │
                         │                  ▼                       │
                         │                 DLQ ◀────────────────────┘
                         │
                         │ (优先级 > 0)
                         └──▶ tasks:pending:priority ──▶ ACTIVE
```

**六个状态说明：**

| 状态 | 说明 | 颜色标识 |
|------|------|---------|
| `PENDING` | 任务已提交，等待执行 | 灰色 |
| `ACTIVE` | 任务正在执行中 | 绿色 |
| `COMPLETED` | 任务执行成功 | 蓝色 |
| `FAILED` | 任务执行失败（可重试） | 橙色 |
| `SCHEDULED` | 任务已调度，等待延迟重试 | 紫色 |
| `DLQ` | 死信队列，重试耗尽的永久失败 | 红色 |

### 2.5 四个队列

| 队列 | Redis 结构 | 用途 |
|------|-----------|------|
| **Primary** | Sorted Set（按优先级排序） | 新任务和重试到期任务 |
| **Retry** | List | 执行失败等待重试的任务 |
| **Scheduled** | Sorted Set（按重试时间排序） | 正在等待重试延迟的任务 |
| **DLQ** | List | 重试耗尽的失败任务（死信） |

### 3.4 核心机制详解

#### 3.4.1 优先级调度

任务使用 Redis Sorted Set 存储，分数为优先级的负值（`-priority`），确保高优先级任务优先被消费。

```
优先级 0（默认）→ 分数 0
优先级 5         → 分数 -5
优先级 10（最高）→ 分数 -10

Sorted Set 按分数升序排列 → -10 < -5 < 0 → 高优先级任务排在前面
```

#### 3.4.2 重试机制

```
任务执行失败
    │
    ├─ 临时错误（429 Too Many Requests / 5xx Server Error）
    │   │
    │   ▼
    │   SCHEDULED（延迟重试）
    │   │
    │   └─ 延迟到期 → 重新进入 PENDING 队列
    │
    └─ 永久错误（4xx Client Error）
        │
        ▼
        DLQ（死信队列，不再重试）
```

- 重试延迟采用**指数退避**策略
- 最大重试次数可配置（默认 3 次）
- 重试耗尽后进入 DLQ，可手动重试

#### 3.4.3 限流器（Rate Limiter）

基于 Redis 的**令牌桶算法**，按任务类型独立限流。

```
配置示例：
  rate_limit_requests = 100
  rate_limit_interval = 60

含义：每 60 秒最多处理 100 个该类型的任务
```

#### 3.4.4 熔断器（Circuit Breaker）

分布式熔断器，防止持续调用已出问题的外部 API：

```
CLOSED（正常）
    │
    ├─ 连续失败达到阈值（默认 10 次）
    │
    ▼
OPEN（断开，拒绝所有请求）
    │
    ├─ 等待重置超时（默认 120 秒）
    │
    ▼
HALF-OPEN（半开，允许少量请求通过）
    │
    ├─ 请求成功 → 恢复 CLOSED
    └─ 请求失败 → 回到 OPEN
```

#### 3.4.5 SSRF 防护

Worker 在发起外部 API 调用前会验证 URL，防止服务端请求伪造（SSRF）攻击。

#### 3.4.6 回调通知（Webhook）

任务支持 `callback_url` 参数，任务完成后会向该 URL 发送 POST 请求：

```json
{
  "task_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "state": "COMPLETED",
  "result": "翻译结果...",
  "http_status": 200,
  "completed_at": "2026-05-20T10:00:05.000Z"
}
```

### 4.3 页面功能详解

#### 4.3.1 Dashboard — 仪表盘

系统的"驾驶舱"，分为 4 个区域：

**区域一：快速提交任务**
- 任务类型下拉框（动态加载可用类型）
- 内容输入框
- 参数输入框（JSON 格式，可选）
- Execute 按钮
- 提交成功后显示任务 ID

**区域二：队列统计（QueueStats）**
- 4 个队列深度卡片：PRIMARY（蓝色）、RETRY（琥珀色）、SCHEDULED（紫色）、DLQ（红色）
- 状态分布条形图

**区域三：任务流程图（TaskFlowGraph）**
- 基于 ReactFlow 的交互式流程图
- 8 个节点展示完整任务流转
- 点击节点可跳转到对应任务列表
- 支持拖拽和缩放

**区域四：Worker 状态（WorkerStats）**
- Worker 总数、健康数、过期数
- 每个 Worker 的状态、熔断器状态、成功/失败计数
- 每 30 秒自动更新

**实时更新：** 通过 SSE 连接 `/api/v1/queues/status/stream`，实时接收队列状态变化。

#### 4.3.2 TaskTypeManager — 任务类型管理

- **列表展示**：表格显示所有任务类型
- **注册新类型**：多段式表单对话框
  - BASIC_INFO：type_id、name、description
  - API_CONFIG：api_base_url、api_endpoint、http_method
  - AUTHENTICATION：认证方式选择
  - TEMPLATE_AND_RESPONSE：请求模板、响应提取
  - RETRY_AND_RATE_LIMIT：重试、限流配置
- **编辑**：复用注册表单（预填数据）
- **测试**：发送真实 API 请求测试配置
- **激活/停用**：Toggle 按钮
- **删除**：确认后删除

#### 4.3.3 TasksHistory — 任务历史

- **搜索**：按任务 ID 模糊搜索
- **筛选**：状态下拉框、类型下拉框
- **分页表格**：TASK_ID / STATUS / TYPE / CREATED / ACTIONS
- **展开详情**：
  - 任务信息（类型、内容、结果）
  - 时间信息（创建、更新、完成、耗时）
  - 重试信息（当前次数、最大次数、下次重试时间）
  - 附加参数
  - 状态变更历史
  - 错误记录
  - 任务结果数据

#### 4.3.4 TasksCleanup — 清理管理（规划中）

- 数据清理：删除已完成的旧任务
- DLQ 管理：查看和管理死信队列
- 批量操作：批量清理操作

### 4.4 国际化（i18n）

- **实现方式**：React Context + `useI18n()` Hook + `t()` 翻译函数

## 五、异步能力详解

### 5.1 三大异步能力

| 能力 | 异步机制 | 接口 |
|------|---------|------|
| 任务提交 | 提交即返回，后台异步处理 | `POST /api/v1/tasks/submit` |
| 状态捕获 | SSE 实时推送 + 回调通知 + 轮询 | `GET /api/v1/queues/status/stream` |
| 结果获取 | 轮询查询 + 回调通知 | `GET /api/v1/tasks/{task_id}` |

### 5.2 异步任务提交

```
客户端 --POST--> API 服务 --LPUSH--> Redis 队列 --返回 task_id--> 客户端
                                          |
                                     Worker 异步消费
```

提交接口是**非阻塞**的：API 只负责将任务写入 Redis 队列，立即返回 `task_id`。实际的 HTTP 调用由后台 Worker 异步执行。

### 5.3 三种状态捕获方式

#### 方式一：SSE 实时推送（推荐）

```
API 服务 --订阅--> Redis Pub/Sub 频道 "queue-updates"
                      ^
Worker 状态变更时 ------+
```

SSE 事件类型：

| 事件类型 | 触发时机 | 说明 |
|---------|---------|------|
| `initial_status` | 连接建立时 | 队列深度 + 状态统计 |
| `task_state_changed` | 任务状态变更时 | 任务 ID + 新旧状态 + 队列深度 |
| `task_created` | 新任务提交时 | 任务 ID + 类型 |
| `heartbeat` | 每 30 秒 | 时间戳（保活） |
| `error` | 异常时 | 错误信息 |

#### 方式二：回调通知（Webhook）

提交任务时指定 `callback_url`，任务完成后系统自动向该 URL 发送 POST 请求。

#### 方式三：轮询查询

客户端定时调用 `GET /api/v1/tasks/{task_id}` 检查任务状态。

### 5.4 三种方式对比

| 特性 | SSE 实时推送 | 回调通知 | 轮询查询 |
|------|-------------|---------|---------|
| **实时性** | 毫秒级 | 秒级 | 秒~分钟级 |
| **连接方式** | 长连接 | 无连接 | 短连接 |
| **服务端资源** | 中（维持连接） | 低 | 低 |
| **客户端复杂度** | 中 | 低（需暴露端口） | 低 |
| **适用场景** | 前端实时 UI | 微服务间通信 | 简单脚本 |
| **状态粒度** | 所有状态变更 | 仅完成时 | 按需查询 |

## 六、数据存储与持久化

### 6.1 存储结构

任务数据存储在 **Redis Hash** 中，Key 模式为 `task:{task_id}`。

| 字段 | 说明 |
|------|------|
| `state` | 任务状态（PENDING/ACTIVE/COMPLETED/FAILED/SCHEDULED/DLQ） |
| `task_type` | 任务类型 ID |
| `content` | 任务内容 |
| `result` | 任务执行结果 |
| `http_status` | 外部 API 返回的 HTTP 状态码 |
| `created_at` | 创建时间 |
| `updated_at` | 最后更新时间 |
| `completed_at` | 完成时间 |
| `started_at` | 开始执行时间 |
| `worker_id` | 处理该任务的 Worker |
| `retry_count` | 当前重试次数 |
| `max_retries` | 最大重试次数 |
| `error_history` | 错误历史记录（JSON 数组） |
| `state_history` | 状态变更历史（JSON 数组） |

> **注意：** 项目不使用 Celery 自带的结果存储（`backend=None`），而是完全依赖自定义的 `task:{task_id}` Redis Hash。

### 6.2 持久化策略

采用 **RDB + AOF 双重持久化**，确保数据安全：

#### RDB 快照（定期全量备份）

| 触发条件 | 说明 |
|---------|------|
| `--save 900 1` | 900 秒内至少 1 次写入则触发快照 |
| `--save 300 10` | 300 秒内至少 10 次写入则触发快照 |
| `--save 60 10000` | 60 秒内至少 10000 次写入则触发快照 |

- **优点**：文件紧凑，恢复速度快
- **缺点**：快照之间有数据丢失风险

#### AOF 追加（实时增量日志）

| 参数 | 值 | 说明 |
|------|-----|------|
| `--appendonly yes` | 已开启 | 启用 AOF 持久化 |
| `--appendfsync everysec` | 每秒同步 | 最多丢失 1 秒数据 |

- **优点**：数据安全性高，最多丢失 1 秒数据
- **缺点**：文件较大，恢复速度较慢

#### 双重保障

```
Redis 写操作
    │
    ├─→ RDB 快照（定期生成 dump.rdb）
    │
    └─→ AOF 日志（实时追加 appendonly.aof）

Redis 重启恢复时：
    1. 优先加载 AOF（数据更完整）
    2. 若无 AOF 则加载 RDB
```

### 6.3 TTL 过期策略

| 任务状态 | TTL | 说明 |
|---------|-----|------|
| `PENDING` | 无 | 永久保留，直到被执行 |
| `ACTIVE` | 无 | 永久保留，直到执行完成 |
| `SCHEDULED` | 无 | 永久保留，直到重试 |
| `COMPLETED` | 24 小时 | 可通过 `TASK_RESULT_TTL` 配置 |
| `DLQ` | 24 小时 | 可通过 `TASK_RESULT_TTL` 配置 |

### 6.4 数据安全保证

| 保障级别 | 机制 | 数据丢失风险 |
|---------|------|-------------|
| **高** | RDB + AOF 双重持久化 | 最多丢失 1 秒数据 |
| **中** | Docker 数据卷 | 容器重启不丢数据 |
| **低** | 仅 RDB | 可能丢失最近一次快照后的数据 |

```yaml
    ports: ["6379:6379"]

      --save 900 1 --save 300 10 --save 60 10000

      replicas: 3  # 可配置 Worker 数量
    depends_on: [redis]

    depends_on: [redis]
```

```bash
curl -X POST http://localhost:8000/api/v1/task-types/ \
  -H "Content-Type: application/json" \
  -d '{
    "type_id": "translate",
    "name": "文本翻译",
    "api_base_url": "https://api.mymemory.translated.net",
    "api_endpoint": "/get",
    "http_method": "GET",
    "response_jsonpath": "$.responseData.translatedText",
    "max_retries": 3,
    "enabled": true
  }'
```

```bash
curl -X POST http://localhost:8000/api/v1/tasks/submit \
  -H "Content-Type: application/json" \
  -d '{
    "task_type": "translate",
    "content": "Hello, world!",
    "params": {"target_lang": "zh"}
  }'
# 返回: {"task_id": "xxx", "state": "PENDING"}
```

```bash
# SSE 实时监听
curl -N http://localhost:8000/api/v1/queues/status/stream
```

### 10.4 第四步：获取结果

```bash
curl http://localhost:8000/api/v1/tasks/xxx
```
